# Technical Story: Implement Decimal-to-Integer (Money Pattern) Data Migration and Precision Arithmetic in SQLite

## 1. Context & Problem Statement

Currently, the system uses floating-point numbers (`Float`) in Prisma and SQLite to handle financial values (`amount` and `baseAmountUsd`). Due to the IEEE 754 standard, double-precision floats introduce non-deterministic rounding errors during mathematical aggregations (e.g., $0.1 + 0.2 = 0.30000000000000004$), causing cumulative balance mismatches across financial accounts.

Additionally, SQLite lacks a native fixed-point `DECIMAL` type. To achieve absolute mathematical precision and prevent "penny leakages", the architecture must transition to the **Money Pattern**, storing currency amounts as standard integers (`Int`) representing the smallest fractional unit (cents / multiplied by 100).

---

## 2. Technical Specification & Architecture Requirements

### 2.1. Schema Mutation (`schema.prisma`)

* Alter the fields `amount` and `baseAmountUsd` in the `Transaction` (or equivalent) model from `Float` to `Int`.
* Run `prisma migrate dev` to generate the migration file, but **do not apply it blank**; it must contain the data preservation script specified in section 3.

### 2.2. Ingestion Pipeline Layer (`ExecuteImportUseCase`)

* Modify the data parser to intercept raw float/string values from the Excel/CSV engine.
* Every transaction amount must be multiplied by `100` and passed through `Math.round()` to ensure a clean integer representation before the bulk insert.
* *Example:* An expense of `$1,530.45` must be persisted in SQLite strictly as `153045`.

### 2.3. Query & Balance Domain Logic

* Refactor the calculation endpoint/use-case (e.g., `GET /api/accounts/balances`).
* Apply strict transactional polarity rules when computing running totals in memory or database queries:

$$\text{Account Balance} = \sum \text{INCOME} - \sum \text{EXPENSE} + \sum \text{TRANSFER\_IN} - \sum \text{TRANSFER\_OUT}$$


* The application must deduct the transfer amount specifically from the `senderAccountId` and credit it to the `receiverAccountId`.

### 2.4. Presentation Layer (Next.js Frontend Components)

* React components must **never** perform raw float operations.
* Keep the data as integers during any client-side accumulation.
* Divide the final total by `100` (e.g., `(amount / 100).toFixed(2)`) **only and exclusively** inside the JSX rendering block for visual representation to the user.

---

## 3. Data Migration Script (SQLite Compatible)

To prevent historical data loss, the Prisma migration SQL file must execute an explicit type cast and scaling step *before* locking the table columns to integer restrictions:

```sql
-- PRAGMA foreign_keys=OFF;
-- 1. Create a deployment fallback backup or temporary table if Prisma optimization requires it.
-- 2. Scale existing float amounts to cents integers:
UPDATE "Transaction" 
SET 
  "amount" = CAST(ROUND("amount" * 100) AS INTEGER),
  "baseAmountUsd" = CAST(ROUND("baseAmountUsd" * 100) AS INTEGER);

-- 3. Apply the structural change to INT (handled natively via Prisma Migration engine mapping).

```

---

## 4. Acceptance Criteria (Definition of Done)

1. **Zero Floats in DB:** The updated `schema.prisma` contains no `Float` types for currency representations.
2. **Idempotency Integrity:** The `importHash` generation remains stable. If the original hash algorithm relied on the raw string/float representation from the source file (e.g., `toFixed(2)`), the parser must compute the hash *before* scaling the value to an integer, preventing duplicate entries upon re-importation.
3. **Exact Matching Balances:** Summing account transaction history via the API yields exact mathematical matches with the real-world account balances.
4. **Valid Edge Cases:** Transactions with zero value (`0.00`) or specific decimal configurations (e.g., values ending in `.50`, `.05`, `.99`) parse correctly without dropping decimals or adding phantom cents.
